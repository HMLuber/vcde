Was ist Echtzeit-Objekterkennung?

Aufbauend auf Vorlesungsfolie: „Erkennung: Klassifikation, Segmentierung, Objekterkennung, Anomalieerkennung“.

Objekterkennung löst gleichzeitig zwei Aufgaben: Lokalisierung (wo ist das Objekt? → Bounding Box) und Klassifikation (was ist es? → Klasse + Konfidenz). Echtzeit bedeutet mindestens 25–30 Frames pro Sekunde – jeder Frame muss in unter 33 ms verarbeitet sein.

Im Gegensatz zu klassischen Bildanalyseverfahren ermöglicht sie die gleichzeitige Erkennung mehrerer Objekte und deren Klassifizierung in wenigen Millisekunden. Die Technologie nutzt tiefe neuronale Netze (Deep Learning), um visuell Muster zu erkennen und präzise Bounding Boxes um Objekte zu zeichnen – Grundlage für autonomes Fahren, Videoüberwachung, medizinische Bilddiagnose und industrielle Qualitätsprüfung.

📦

Lokalisierung

Position und Größe jedes Objekts als Bounding Box (x, y, Breite, Höhe).

🏷️

Klassifikation

Gleichzeitige Zuweisung eines Klassen-Labels mit Konfidenzwert.

⚡

Echtzeitfähigkeit

One-Stage-Detektoren wie YOLO erreichen >30 FPS auf Consumer-Hardware.

Begriffserklärungen
Begriff 01
Bounding Box

Achsenparalleles Rechteck um ein erkanntes Objekt – definiert durch Mittelpunkt (cx, cy) sowie Breite und Höhe.

Begriff 02
Confidence Score

Zahl [0–1]: Wie sicher ist das Modell, dass die Box ein Objekt enthält und die Klasse korrekt ist?

Begriff 03
IoU – Intersection over Union

Schnittfläche / Vereinigungsfläche zweier Boxen. 0 = keine Überlappung, 1 = identisch. COCO-Schwellwert: ≥ 0,5.

Begriff 04
NMS – Non-Maximum Suppression

Bereinigt mehrfach erkannte Objekte: Beste Box behalten, alle Nachbarboxen mit IoU > 0,45 unterdrücken.

Begriff 05
Anchor Box

Vordefinierte Box-Formen als Vorhersage-Referenz. YOLOv8 ist ankerfrei und sagt den Mittelpunkt direkt vorher.

Begriff 06
mAP – mean Average Precision

Hauptmetrik für Objektdetektoren: Durchschnitt der Average Precision über alle Klassen bei variierenden IoU-Schwellwerten.

Begriff 07
CNN – Convolutional Neural Network

Neuronales Netzwerk spezialisiert auf räumliche Daten wie Bilder. Extrahiert Merkmale automatisch durch Faltungsoperationen – von einfachen Kanten bis zu komplexen Formen.

Kapitel 2
Motivation

Die Fähigkeit, Objekte augenblicklich und präzise zu erkennen, ist in einer zunehmend vernetzten und autonomen Welt unverzichtbar. Herkömmliche Bildanalyse-Methoden sind zu langsam und fehleranfällig für moderne Anforderungen.

Warum ist Geschwindigkeit so kritisch? Ein autonomes Fahrzeug, das einen Fußgänger 200 ms zu spät erkennt, hat bei 50 km/h bereits 2,8 m zurückgelegt – zu viel für eine sichere Reaktion. Millisekunden entscheiden hier über Leben und Tod, und das gilt nicht nur im Straßenverkehr: Industrieroboter, die Defekte am Fließband erkennen, medizinische Systeme, die während einer Operation assistieren, oder Sicherheitskameras, die Eindringlinge in Echtzeit melden – all diese Anwendungen verlangen, dass jeder einzelne Frame innerhalb von unter 33 ms vollständig verarbeitet ist.

Der Bedarf wächst rasant: Der globale Markt für Computer Vision wird bis 2030 auf über 50 Milliarden US-Dollar geschätzt. Was vor zehn Jahren noch spezialisierte Rechenzentren erforderte, läuft heute auf Smartphones, Mikrocontrollern und direkt im Browser – und macht Echtzeit-Objekterkennung damit zu einer der praktisch relevantesten Technologien unserer Zeit.

🐢 Two-Stage – langsam

Faster R-CNN erzielt hohe Genauigkeit (~42 mAP), benötigt aber 100–500 ms pro Frame – für Live-Video nicht einsetzbar.

⚡ One-Stage – schnell

YOLOv8n erreicht 37 mAP bei unter 5 ms pro Frame auf der GPU, im Browser 10–30 FPS – Echtzeit ganz ohne Server.

Kapitel 3
Technik

Technische Details basieren auf der Vorlesungsfolie „Objekterkennung” und der behandelten Objekterkennungspipeline.

Dieses Kapitel beleuchtet den vollständigen technischen Weg vom rohen Pixel zur erkannten Bounding Box: Wie arbeitet ein Convolutional Neural Network? Was tut jede Stufe der YOLO-Pipeline? Wie entscheidet das Netz, ob und wo ein Objekt ist? Und wie läuft das ganze Modell direkt im Browser – ohne Server?

3.1 · Convolutional Neural Networks – Wie Netze Merkmale lernen

Der Kern aller modernen Objektdetektoren ist ein Convolutional Neural Network (CNN). Im Gegensatz zu klassischen vollverbundenen Netzen teilt eine Faltungsschicht dieselben Gewichte über die gesamte räumliche Ausdehnung des Bildes – dadurch lernt das Netz Merkmale, die verschiebungsinvariant sind: Eine Katze in der linken Ecke aktiviert denselben Filter wie eine Katze in der Mitte.

⊞

Faltung (Conv)

Ein kleiner Filter (z.B. 3×3) gleitet über das Bild und berechnet pro Position ein gewichtetes Skalarprodukt. Ergebnis: eine Feature Map mit erkannten lokalen Mustern.

↘

Pooling

Max-Pooling (2×2) halbiert die räumliche Auflösung und behält nur den stärksten Aktivierungswert pro Region. Spart Rechenzeit und macht das Netz robuster gegen kleine Verschiebungen.

⚡

Aktivierung (ReLU)

Rectified Linear Unit: f(x) = max(0, x). Bringt Nichtlinearität ins Netz, ohne viel Rechenaufwand. Ohne Aktivierungsfunktionen wäre das Netz nur eine lineare Transformation.

🏛️

Tiefe Hierarchie

Frühe Schichten: Kanten und Farben. Mittlere Schichten: Texturen und Formen. Späte Schichten: Semantische Konzepte wie „Gesicht” oder „Reifen”. Jede Schicht baut auf der vorherigen auf.

📐
Faltung konkret: Ein 3×3-Filter mit 32 Ausgabekanälen erzeugt aus einem 640×640-Eingabebild eine Feature Map von 640×640×32. Nach Batch-Normalisierung und ReLU folgt meist ein weiterer Conv-Block. Stride 2 (Filter springt je 2 Pixel) halbiert die Auflösung – das ist effizienter als separates Pooling und wird in modernen Netzen bevorzugt.
3.2 · Vorverarbeitung – Letterboxing und Normalisierung

Bevor ein Frame ins Netz geht, muss er auf die Eingabegröße des Modells gebracht werden – bei YOLOv8 standardmäßig 640×640 Pixel. Naives Skalieren würde das Seitenverhältnis verzerren und Objekte stauchen oder strecken, was die Erkennungsgenauigkeit drastisch verschlechtert. Die Lösung heißt Letterboxing:

A

Originalbild

Beliebiges Seitenverhältnis (z.B. 1920×1080 → 16:9)

›
B

Proportional skalieren

Längste Seite auf 640 px bringen, andere Seite entsprechend kürzer

›
C

Padding hinzufügen

Grauer Rand (RGB 114,114,114) oben/unten oder links/rechts auffüllen

›
D

Normalisieren

Pixelwerte [0–255] → [0.0–1.0], Kanäle CHW-Format (3×640×640)

📌
Warum ist das Padding-Offset wichtig? Nach der Inferenz müssen die vorhergesagten Bounding-Box-Koordinaten zurücktransformiert werden: erst Padding abziehen, dann durch den Skalierungsfaktor teilen. Passiert das nicht, sind alle Boxen um den Padding-Versatz verschoben – eine häufige Fehlerquelle bei eigenen Implementierungen.
3.3 · Die YOLO-Pipeline – Input → Netzwerk → Bounding Boxes

Gitter-Struktur (Grid) Das Bild wird konzeptuell in ein S×S-Raster unterteilt. Bei YOLOv8 entstehen durch den Backbone drei Feature Maps auf verschiedenen Auflösungen: 80×80, 40×40 und 20×20 – also 8.400 potenzielle Gitterzellen insgesamt. Jede Zelle ist für Objekte verantwortlich, deren Mittelpunkt in ihr liegt. Für jede Zelle werden gleichzeitig Bounding Boxes, Confidence Score und Klassenwahrscheinlichkeiten vorhergesagt.
3.4 · Backbone: CSPDarknet – der Merkmals-Extraktor

Der Backbone ist das „Sehen” des Netzes. Er nimmt das 640×640-Bild entgegen und erzeugt kompakte, semantisch reiche Feature Maps. YOLOv8 verwendet CSPDarknet (Cross-Stage Partial Network), eine effiziente Variante tiefer Residualnetze.

Residual-Verbindungen (Skip Connections)

Inspiriert von ResNet: Statt nur den Ausgang einer Schicht weiterzugeben, wird dieser mit dem Eingang addiert (y = F(x) + x). Das löst das Vanishing Gradient Problem und ermöglicht sehr tiefe Netze (50+ Schichten), ohne dass der Gradient beim Training verschwindet.

CSP-Blöcke (Cross-Stage Partial)

Der Eingang wird in zwei Hälften geteilt: eine Hälfte durchläuft die Residualblöcke, die andere wird direkt weitergeleitet und am Ende konkateniert. Vorteil: Der Gradient fließt auf zwei Wegen zurück → weniger redundante Berechnungen, weniger Speicherbedarf, ~20 % schneller als Standard-ResNet bei gleicher Genauigkeit.

P3

Small Objects

Feature Map 80×80 – hohe räumliche Auflösung für kleine Objekte (z.B. Fußgänger im Hintergrund, Vögel).

P4

Medium Objects

Feature Map 40×40 – mittlere Detailstufe, ausgewogenes Verhältnis aus Semantik und Lokalisation.

P5

Large Objects

Feature Map 20×20 – stark komprimiert, hohe Semantik für große nahe Objekte (z.B. Fahrzeug im Vordergrund).

3.5 · Neck: FPN + PAN – Multi-Scale Feature Fusion

Der Backbone liefert Feature Maps auf drei Skalen (P3, P4, P5). Das Problem: tiefe Schichten sehen was (hohe Semantik), aber nicht mehr genau wo (geringe Auflösung). Flache Schichten sehen wo, aber nicht was. Der Neck kombiniert beides durch zwei komplementäre Informationspfade.

FPN – Feature Pyramid Network (Top-Down)

Semantische Information fließt von tiefen zu flachen Schichten: P5 wird upgesampelt und mit P4 addiert → P4 erhält semantischen Kontext. Dann wird das Ergebnis upgesampelt und mit P3 addiert → P3 erhält ebenfalls semantischen Kontext. Alle drei Ebenen wissen jetzt was zu sehen ist.

PAN – Path Aggregation Network (Bottom-Up)

Lokalisierungsinformation fließt zurück von flachen zu tiefen Schichten: P3 wird downgesampelt und mit P4 kombiniert, dann weiter zu P5. So wissen die tiefen Schichten auch wieder genau wo sich ein Objekt befindet. Der kombinierte FPN+PAN-Neck macht YOLO robust über alle Objektgrößen.

🔀
Warum Multi-Scale so wichtig ist: Ein Fußgänger 50 m entfernt belegt vielleicht 20×40 Pixel im Frame – auf der P5-Feature-Map (20×20) nur noch ~0,6×1,3 Zellen, also praktisch unsichtbar. Auf P3 (80×80) hat das selbe Objekt ~2,5×5 Zellen – genug für eine präzise Detektion. Ohne Multi-Scale würde YOLO kleine Objekte zuverlässig übersehen.
3.6 · Detection Head – ankerfrei und entkoppelt

Der Head nimmt die gefusionten Feature Maps und trifft die finale Vorhersage. YOLOv8 verwendet einen decoupled head: separate Zweige für Bounding-Box-Regression und Klassifikation – das verbessert die Konvergenz beim Training.

📦

BBox-Regression

Für jede Gitterzelle werden 4 Werte vorhergesagt: Δx, Δy (Offset des Mittelpunkts von der Zellmitte) sowie Breite und Höhe als Distribution Focal Loss (DFL) – robuster als frühere direktte Regression.

🏷️

Klassifikation

Parallel zur BBox-Branch: Für jede Gitterzelle wird ein Sigmoid über 80 Klassen berechnet (nicht Softmax!) – ein Objekt kann mehreren Klassen gleichzeitig angehören, z.B. „person” + „athlete”.

⚓

Ankerfrei (Anchor-Free)

YOLOv1–v5 nutzen vordefinierten Anchor-Boxen als Referenzform. YOLOv8 sagt den Box-Mittelpunkt direkt vorher (Task-Aligned Assigner) – weniger Hyperparameter, einfacheres Training, bessere Performance.

Head-Ausgabe pro Feature-Map-Zelle (schematisch) 
# Pro Gitterzelle auf jeder der 3 Skalen:
output = {
    'bbox':  [cx, cy, w, h],          # 4 Werte → Bounding Box
    'score': [p_class_0, ..., p_class_79]  # 80 Klassen-Wahrscheinlichkeiten
}
# Gesamt: (80×80 + 40×40 + 20×20) × (4+80) = 8400 × 84 Werte
# → NMS filtert auf die relevanten Detektionen
3.7 · IoU – Intersection over Union im Detail

IoU ist die zentrale Metrik um zu beurteilen, wie gut eine vorhergesagte Bounding Box mit dem Ground-Truth-Label übereinstimmt. Sie wird sowohl als Trainings-Loss als auch für die Auswertung und NMS verwendet.

Formel

IoU = Schnittfläche / Vereinigungsfläche

= |A ∩ B| / |A ∪ B|
= |A ∩ B| / (|A| + |B| − |A ∩ B|)

Wertebereich: 0 (keine Überlappung) bis 1 (identische Boxen).
COCO-Standard: Detektion gilt als korrekt (True Positive) ab IoU ≥ 0,5.

IoU-Varianten im Training

YOLOv8 verwendet CIoU (Complete IoU) als Trainingsloss – berücksichtigt neben der Überlappung auch Distanz der Mittelpunkte und Seitenverhältnis. Vorteil: Gradient auch dann vorhanden, wenn sich die Boxen nicht überlappen (IoU = 0 → normaler IoU-Loss hätte Gradient 0).

GIoU: Generalized – bestraft fehlende Überlappung
DIoU: Distance – bestraft Mittelpunkt-Distanz
CIoU: Complete – GIoU + DIoU + Aspect Ratio
3.8 · Non-Maximum Suppression – Schritt für Schritt

Nach dem Head liegen für alle 8.400 Gitterzellen Vorhersagen vor – viele davon detektieren dasselbe Objekt mehrfach. NMS reduziert diese auf eine einzige, bestmögliche Box pro Objekt.

①

Filtern nach Confidence

Alle Boxen mit Score < Schwellwert (z.B. 0,40) werden verworfen. Typisch bleiben nur wenige Hundert Kandidaten übrig.

②

Sortieren nach Score

Alle verbleibenden Boxen werden absteigend nach Konfidenz sortiert. Die Box mit dem höchsten Score kommt zuerst.

③

Beste Box wählen

Die Box mit dem aktuell höchsten Score wird als Detektion übernommen und aus der Kandidatenliste entfernt.

④

Überlappende Boxen unterdrücken

Alle verbleibenden Boxen derselben Klasse, deren IoU mit der gewählten Box ≥ 0,45 ist, werden gelöscht – sie zeigen auf dasselbe Objekt.

⑤

Wiederholen

Schritte ③–④ werden wiederholt, bis keine Kandidaten mehr übrig sind. Ergebnis: eine Box pro Objekt, die bestmögliche.

⚠️
NMS-Grenzfälle: Stehen zwei gleiche Objekte sehr nah beieinander (z.B. zwei nebeneinanderstehende Personen), kann NMS eine davon fälschlicherweise unterdrücken. Lösungsansätze: niedrigerer IoU-Schwellwert (0,3–0,4), Soft-NMS (Confidence reduzieren statt Box löschen) oder klassenspezifisches NMS.

WebAssembly (WASM)

Binäres Bytecode-Format, das in jedem modernen Browser läuft – kein Plugin nötig. ONNX Runtime kompiliert die Inferenz-Engine zu WASM und nutzt SIMD-Instruktionen für vektorisierte Berechnungen. Ergebnis: 10–25 FPS auf typischer Consumer-CPU, vollständig offline-fähig.

WebGPU (wo verfügbar)

Neue Browser-API für direkte GPU-Programmierung im Web – deutlich schneller als WebGL für Matrixmultiplikationen. ONNX Runtime Web erkennt automatisch, ob WebGPU verfügbar ist, und nutzt es bevorzugt. Auf modernen Laptops: 30–60 FPS erreichbar. Fallback auf WASM wenn nicht unterstützt.

🧵
Web Worker für nicht-blockierende Inferenz: Die ONNX-Inferenz läuft in dieser Demo in einem separaten Web Worker – einem JavaScript-Thread, der unabhängig vom UI-Thread arbeitet. Dadurch friert die Seite nicht ein, während das Netz rechnet. Der Haupt-Thread übernimmt nur Preprocessing und Rendering (Canvas), der Worker ausschließlich die ONNX-Inferenz. Kommunikation per postMessage() mit transferierbaren ArrayBuffers (zero-copy, kein Kopieren des Float32-Tensors).
3.10 · Klassische Pipeline: OpenCV + YOLO (Python)

In produktiven Server-Anwendungen kombiniert man OpenCV für Video-Capture, Farbkonvertierung (BGR→RGB), Letterboxing und Visualisierung mit der Ultralytics-YOLO-API für die Inferenz. Die Pipeline ist nahezu identisch zur Browser-Demo – nur Sprache und Laufzeitumgebung unterscheiden sich.

Diese Implementierung ergänzt die Vorlesungsfolie „Objekterkennung” um eine browserseitige Alternative zu OpenCV.

OpenCV + YOLO (Python) 
import cv2
from ultralytics import YOLO

model = YOLO('yolov8n.pt')
cap   = cv2.VideoCapture('video.mp4')  # oder 0 für Webcam

while cap.isOpened():
    ret, frame = cap.read()
    if not ret: break
    results = model.predict(frame, conf=0.30)
    cv2.imshow('YOLO', results[0].plot())
    if cv2.waitKey(1) == ord('q'): break
🐍

Python + GPU

CUDA-Beschleunigung über PyTorch. Auf einer NVIDIA RTX 3080: ~200–400 FPS bei YOLOv8n. Ideal für Batch-Verarbeitung und Forschung.

🌐

Browser (diese Demo)

ONNX Runtime Web + WebAssembly. 10–30 FPS ohne Server, direkt auf dem Client. Ideal für Demos, Privatsphäre-sensitive Anwendungen.

📱

Edge / Mobile

TFLite oder CoreML auf Smartphone. YOLOv8n läuft auf modernen Smartphones mit 30+ FPS – ohne Cloud-Anbindung, ohne Latenz.
